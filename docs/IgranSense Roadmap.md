# IgranSense – Realistic MVP Roadmap, Task Assignments & Pitch Narrative

## Executive Summary

This document provides a realistic, budget-conscious roadmap for IgranSense, a solar-powered edge micro data center platform for precision agriculture in low-connectivity regions of Morocco. It is tailored to the founding team — three 1337 Network students with complementary skills and no hardware budget — and is structured to maximize what can be achieved with free tools, open data, and software-only prototyping. The roadmap covers: individual task assignments for each co-founder, a realistic 12-month plan aligned with UM6P Launchpad milestones (M3/M6/M9/M12), and a pitch narrative designed to resonate with any jury.

***

**Core responsibility:** Design the edge computing layer, system architecture, and everything related to how data flows from sensors through the MDC to the dashboard.

**Individual tasks (pre-pitch, now → March 3):**

1. **Architecture diagram (Excalidraw or draw.io):**

   - Draw the **current MVP** architecture: Mock data files → Local Python/Node service → Dashboard.

2. **Edge simulation service:**
   - Build a lightweight local service (Python preferred — FastAPI or Flask) that:
     - Reads mock sensor JSON files.
     - Applies simple rule-based logic (thresholds for irrigation alerts).
     - Serves processed data to the frontend via a REST API.
   - This simulates what the real MDC would do: ingest, process, and serve decisions locally.

3. **Scientific basis research:**
   - Read and summarize in your own words 2–3 key references on edge computing in agriculture:
     - A 2025 study demonstrated that LoRaWAN + edge computing reduces irrigation latency from hours (cloud) to seconds (edge), cuts bandwidth by over 90%, and enables autonomous operation on solar power.[^1]
     - An IoT gateway study showed that edge-computing gateways with LoRaWAN achieve low cost, low latency, and low power consumption for agricultural monitoring.[^2]
     - Cloud-edge-device collaborative computing is emerging as a transformative paradigm for smart agriculture, addressing limitations of cloud-only architectures in distributed IoT networks.[^3]
   - Prepare 3–4 bullet points for the TRL/Scientific Basis slide.

4. **TRL positioning statement:**
   - Current TRL: **3** — Concept formulated, architecture designed, individual components (LoRaWAN, SBCs, sensors) validated in existing literature, software proof-of-concept with synthetic data in progress.[^4][^5][^6]
   - Target after Launchpad M6: **TRL 4–5** — Integrated prototype validated in lab then tested in relevant farm environment.[^5]

**Individual tasks (pre-pitch, now → March 3):**

1. **Dashboard MVP — 4 screens:**

     - Data source (which sensor values triggered it).

   **Screen 4: System Status**
   - Mock display of:
     - MDC status (online/offline indicator).

     - Sensor node battery levels.

### Zakariya El Kharroubi — IoT & Data Layer Lead

**Core responsibility:** Design the sensor/IoT layer, generate realistic mock data, and own the agronomic logic that drives the system's recommendations.

**Individual tasks (pre-pitch, now → March 3):**

1. **Mock data generation:**
   - Create JSON/CSV files simulating 30 days of sensor data for a 50-hectare tomato farm in the Souss-Massa region:

   **sensors.json** — Sensor registry:
   ```
   Fields: sensor_id, field_id, type (soil_moisture, temperature, humidity, etc.), lat, lng, status
   ```

   **readings.json** — Time-series data:
   ```
   Fields: timestamp (every 30 min), sensor_id, value, unit, quality_flag
   ```

   **ndvi_snapshots.json** — Weekly NDVI values per field:
   ```
   Fields: date, field_id, mean_ndvi, min_ndvi, max_ndvi
   ```

   - Use realistic value ranges:
     - Soil moisture: 8–45% (sandy loam in Souss-Massa).
     - Air temperature: 12–38°C (February–March range).
     - Humidity: 30–80%.
     - NDVI for healthy tomato crop: 0.5–0.85; stressed: 0.25–0.45.

2. **Agronomic rules definition:**
   - Define irrigation thresholds by crop type (start with tomatoes):
     - Soil moisture < 18%: critical — irrigate immediately.
     - Soil moisture 18–25%: warning — schedule irrigation within 24h.
     - Soil moisture > 25%: OK — no action needed.
   - Define stress detection rules:
     - NDVI drops > 15% week-over-week: potential stress alert.
     - NDVI < 0.35 for healthy-stage crop: investigate immediately.
   - Write these rules as a simple JSON config file that Souhail's service reads.

3. **Real data enrichment (free sources):**
   - Pull real weather data for Souss-Massa from Open-Meteo API (free, no key required) to make mock data realistic.[^7][^8]
   - Look up real NDVI ranges from Sentinel-2 satellite data (Copernicus Open Access Hub — free).
   - Research shows that NDVI and soil moisture are "highly sensitive to moisture, making them the best indices for determining the timing and location of irrigation".[^9][^10]

4. **Hardware specification document (for slides, not for buying):**
   - Create a "Target Hardware Spec" document listing what the real system would use:
     - MDC: Raspberry Pi 5 / NVIDIA Jetson Nano, IP65 enclosure, 100W solar panel, 100Ah LiFePO4 battery.
     - Sensors: capacitive soil moisture (e.g., SEN0308), DHT22 (temp/humidity), LoRa SX1276 modules.
     - Gateway: RAK7268 LoRaWAN gateway or similar.
     - Estimated BoM cost per unit: aim for <3,000 USD for MVP (this is what Launchpad funding would cover).
   - This shows the jury you have researched the path from software prototype to real hardware.

5. **Stakeholder outreach (low-cost validation):**
   - Talk to 1–2 people (even by phone/WhatsApp):
     - An agronomy student or professor at UM6P or IAV.
     - A farmer or cooperative manager if accessible (ask Mouad for introductions).
   - Ask: "What are your biggest frustrations with current irrigation decisions? Would you use a local dashboard that works without internet?"
   - Document 2–3 quotes for the Validation slide.

## Part 3: Realistic 12-Month Roadmap

This roadmap assumes acceptance into UM6P Launchpad with access to prototyping budget, test farms, and mentorship. Goals are deliberately conservative — better to exceed them than fall short.

### M0–M3: Software MVP & Discovery

**Deliverables:**
- Functional dashboard with mock data (already built for the pitch) refined based on mentor/farmer feedback.
- At least 3 farmer/agronomist interviews completed to validate use cases.
- Edge simulation service running on a laptop, demonstrating offline-first analytics.
- Hardware components researched and sourced (quotes obtained, not purchased yet).
- 1 pilot farm identified and relationship established.

**Learning goals:**
- Souhail: Comfortable with Linux server setup, basic networking, REST API development.
- Yahya: Proficient in React, able to build responsive data-driven interfaces.
- Zakariya: Understands LoRaWAN protocol, can read sensor datasheets, familiar with basic agronomic models.

### M4–M6: First Hardware Prototype & Campus Test

**Deliverables:**
- 1 MDC prototype assembled using Launchpad prototyping budget: Raspberry Pi or Jetson Nano + solar panel + battery + LoRa gateway.
- 3–5 sensor nodes deployed on UM6P campus test plot or partner greenhouse.
- Dashboard connected to real sensor data (replacing mock data).
- Basic irrigation threshold alerts validated against real soil conditions.
- First drone flight (if drone access available via UM6P) with NDVI map processed on the MDC.

**Learning goals:**
- Souhail: Can deploy and manage a Linux-based edge server, configure MQTT broker.
- Yahya: Dashboard handles real-time data updates, mobile-responsive.
- Zakariya: Can assemble, flash, and deploy a LoRa sensor node end-to-end.

### M7–M9: First Farm Pilot & Data Collection

**Deliverables:**
- System deployed on 1 real farm (partner identified in M1–M3).
- Minimum 30 days of continuous data collection (sensor + weather).
- Irrigation recommendations compared against farmer's actual practice (manual baseline).
- Preliminary water usage tracking: install flow meter or use farmer's records to compare "with IgranSense" vs "without" decisions.
- Dashboard refined based on farmer feedback (language, usability, alert frequency).
- System uptime tracked: target >90% uptime on solar power.

**Learning goals:**
- All: Comfortable with field deployment, troubleshooting hardware in outdoor conditions.
- Souhail: Can diagnose and fix edge server issues remotely.
- Zakariya: Can replace/recalibrate sensors in the field.

### M10–M12: Validation & Pre-Commercial Preparation

**Deliverables:**
- Preliminary impact numbers (even if modest): "Our system identified X irrigation events that the farmer would have missed" or "Recommendations aligned with agronomist advice Y% of the time."
- Second pilot site started (if feasible) or extended coverage on first site.
- Hardware BOM finalized with cost estimates for a repeatable unit.
- Pricing model validated with 3–5 customer interviews.
- 1 letter of intent or memorandum of understanding from a farm/cooperative.
- Demo Day presentation with real data, real screenshots, and honest results.

**Realistic impact claims at M12:**
- "System operated autonomously on solar power for X days with Y% uptime."
- "Irrigation alerts were validated against agronomist recommendations with Z% agreement."
- "Dashboard used by N farmers/technicians who reported [specific feedback]."

## Part 6: Pitch Narrative

The following is a ready-to-use pitch paragraph designed to open the presentation. It tells the story behind the problem and solution in plain language that works for any jury member — whether they are a tech expert, an agriculture specialist, or a business evaluator.

***

**"Imagine a farmer in the Souss-Massa valley. He has invested in soil sensors and a drone service. He expects real-time insights — when to irrigate, where his crops are stressed, how to save water. But his farm sits in a zone where 4G barely reaches, and when it does, it is slow and expensive. His sensor data is supposed to travel to a cloud server in Europe, get processed, and come back as a recommendation. But the connection drops. Hours pass. Sometimes days. By the time the alert arrives, his tomatoes have already suffered irreversible stress, and thousands of liters of water have been wasted.**

**This is not a hypothetical scenario. Agriculture consumes over 80% of Morocco's water, yet the country is approaching the internationally recognized water poverty threshold of 500 cubic meters per person per year. Morocco's government is investing billions in drip irrigation and water infrastructure. But all of this investment is undermined if the digital layer — the intelligence that tells a farmer when and where to irrigate — depends on an internet connection that does not reliably exist in the field.**

**IgranSense solves this by flipping the model. Instead of sending farm data to the cloud, we bring the cloud to the farm. Our solution is a portable, solar-powered micro data center that sits at the edge of the field. It connects to soil sensors and drone cameras, processes everything locally, and delivers irrigation and stress alerts directly to the farmer's phone — even with zero internet. When connectivity is available, it syncs. When it is not, the farm keeps running.**

**We are three engineering students from 1337 Network. We do not claim to be experts — we claim to be builders who understand systems, software, and sensors, and who are driven to solve a problem that matters for Morocco's future. With Launchpad support, we will turn our working software prototype into a real, field-tested system on a real farm within 12 months."**

***

### Why This Narrative Works

- **Opens with a human story**, not a technology description — connects emotionally with any jury member.
- **Quantifies the problem** with real data (80% water use, 500 m³ threshold) — satisfies the 30% Problem & Market evaluation weight.[^13][^11]
- **Explains the solution in one sentence** ("bring the cloud to the farm") — no jargon, instantly clear.
- **Acknowledges student status honestly** and reframes it as a strength — addresses the 25% Team & Commitment weight by showing self-awareness and drive.
- **Ends with a clear ask** — ties directly to the "Why Launchpad?" slide.

## Part 7: How to Present Your Constraints Positively

### On Having No Hardware Prototype

**Say:** "Our MVP today is a working software prototype that demonstrates exactly what our edge analytics will do — from data ingestion to irrigation alerts. What we need from Launchpad is the prototyping budget to deploy this on real hardware in a real field. We have already researched the components, estimated the costs, and identified the suppliers."

**Do not say:** "We could not afford hardware." Instead, frame it as a deliberate phased approach: software-first validation before committing resources to hardware.

### On Being Students

**Say:** "We are 1337 Network engineers-in-training, which means we learn by building. Our program has no teachers — it is entirely project-based and peer-driven. This is exactly the mindset needed for a startup: figure it out, build it, iterate."

**Do not say:** "We are not experts." Instead, show awareness of what you know, what you need to learn, and that you have a plan for each gap (mentorship through Launchpad, partnership with agronomists like Mouad).

### On the TRL Level

**Say:** "We are at TRL 3 — concept validated, architecture designed, software proof-of-concept with synthetic data. Each individual technology component — LoRaWAN, edge SBCs, soil sensors, NDVI analytics — is proven in published research. Our contribution is the integration and adaptation to Morocco's specific constraints. With Launchpad, we target TRL 5 by month 9."[^1][^4][^5]
